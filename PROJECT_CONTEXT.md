## Project Overview

Invoicer is a premium dark-UI invoicing SaaS for freelancers and small business owners. It allows a single authenticated user per account to create professional invoices, manage clients, maintain a catalog of reusable line items, and manually track payment status — all without accounting knowledge. The app runs entirely in-memory (no database); data is lost on server restart. The product includes a public marketing landing page (`/`) and a full authenticated workspace (`/dashboard`, `/invoices`, `/clients`, `/settings`).

## Authentication

**Mechanism**: Custom opaque token auth with httpOnly cookies. No JWT library used.

**Cookies**:
- `invoicer_access` — short-lived access token (prefix `at_`, 15 min TTL, 24 random bytes base64url)
- `invoicer_refresh` — long-lived refresh token (prefix `rt_`, 7 day TTL, 32 random bytes base64url)
- Both: `httpOnly: true`, `secure: true`, `sameSite: "strict"`, `path: "/"`

**Password hashing**: `crypto.scryptSync(password, "invoicer-salt", 64)` — **IMPORTANT: hardcoded salt, not per-user**

**Session flow**:
1. Login/Register → `issueSession()` creates access + refresh tokens, stored in `store.accessTokens` + `store.refreshTokens`
2. Subsequent requests → `requireAuth(req)` reads `invoicer_access` cookie, finds matching unexpired token in store
3. On 401 → client calls `POST /api/auth/refresh`, which calls `rotateRefreshToken()`: marks old token used, issues new pair
4. Refresh failure → client redirects to `/login`

**Route protection**: `requireAuth(req)` is called at the top of every authenticated API handler. There is **no middleware** — protection is per-handler only.

**Password reset**: `POST /api/auth/forgot-password` creates a token stored in memory but **never sends an email**. Reset link would need to be constructed manually as `/reset-password?token={rawToken}`.

## State Management

No external state management library. All state is local component state via React hooks.

**Pattern**: `useState` + `useEffect` for data fetching. Each page fetches its own data on mount.

**Toast**: Single React Context (`ToastContext`) for cross-component toast notifications. Provided by `ToastProvider` in `app/(app)/layout.tsx`.

**Auth state**: Not stored in React state. Auth is determined server-side via `requireAuth()`. Client detects 401 responses and attempts refresh, then redirects to login.

**Data flow**:
1. Page mounts → `useEffect` calls `requestJson()` → sets local state
2. Mutations → call `requestJson()` with method POST/PATCH/DELETE → update local state with response
3. Errors → stored in local `error` state, shown inline or via toast

**Client-side API helper** (`app/_lib/api.ts`):
- `requestJson<T>(url, init?, _retry?)` — wraps fetch with credentials, handles 401 refresh
- Deduplicates concurrent refresh calls with a module-level `refreshing` promise variable
- `toErrorMessage(error)` — extracts message from `ApiClientError` or generic `Error`

**AsyncState pattern** used in some pages:
```typescript
type AsyncState<T> = { loading: boolean; data: T | null; error: string | null }
function initialState<T>(): AsyncState<T> // returns { loading: true, data: null, error: null }
```

## Known Issues

1. **No data persistence**: The in-memory store (`globalThis.__invoicer_store__`) is wiped on every server restart or Vercel cold start. All user data is lost.

2. **Hardcoded password salt**: `lib/auth.ts:26` — `crypto.scryptSync(password, "invoicer-salt", 64)` uses a constant salt for all users. Passwords should use unique per-user salts.

3. **Email not implemented**: `POST /api/auth/forgot-password` stores a reset token in memory but never sends an email. The password reset flow is entirely broken — users can't receive the reset link.

4. **Logo upload is a stub**: `POST /api/profile/logo` reads the uploaded file, validates it, then stores a fake URL (`https://cdn.invoicer.local/logos/...`) and discards the file bytes. No real file storage is implemented.

5. **No rate limiting**: The login endpoint has no rate limiting. The frontend handles `TOO_MANY_REQUESTS` error codes, but the backend never returns them.

6. **No auth middleware**: Protected routes are secured by calling `requireAuth()` inside each handler. There is no Next.js middleware (`middleware.ts`) to redirect unauthenticated users at the routing level.

7. **Frontend type mismatch**: `app/_lib/types.ts::Invoice` has fields (`clientName`, `clientEmail`, `businessName`, `businessLogoUrl`, `paymentTerms`, `payments[]`, `sortOrder` on LineItem) not returned by the API. Frontend pages that use `ApiInvoice` (defined locally in `[id]/page.tsx`) are more accurate.

8. **Settings form is partial**: `app/(app)/settings/page.tsx` only binds `addressLine1` (not city, state, postalCode, country) from the address object. Saving will only partially update the address.

9. **In-memory store is not thread-safe**: Under concurrent requests the in-memory arrays could have race conditions (e.g., duplicate invoice numbers, concurrent catalog count checks).

10. **`profile/logo` route fetches logo with safeFetch during PDF generation**: The logo URL is always `cdn.invoicer.local` (a fake domain). `safeFetch` will fail DNS resolution but the error is silently caught. PDF still generates without logo.

## What Is NOT Built

- **Email delivery**: No email service integrated. Reset tokens are generated but never sent. Invoice "send" marks status but doesn't email the client.
- **Database / persistence**: Data is lost on restart. No Postgres, SQLite, or any persistent store.
- **Real file storage**: Logo upload is a stub. No S3, R2, or local file storage implemented.
- **Auth middleware**: No `middleware.ts` — unauthenticated users accessing `/dashboard` etc. get API errors rather than a redirect to login.
- **Rate limiting**: No protection against brute-force on login/register.
- **Multi-user / team accounts**: Single user per account only.
- **Invoice emailing**: The send flow changes status but doesn't actually deliver the invoice to the client.
- **Proper PDF rendering**: The PDF is minimal plain-text (no logo, no layout, no line item table, no branded formatting).
- **Currency conversion**: Dashboard only shows invoices matching the default currency; multi-currency aggregation is not done.
- **Advanced reporting / analytics**: Mentioned in pricing copy as roadmap.
- **About / Contact / Privacy Policy / Terms of Service pages**: Footer links go to `#` (no pages).
- **Per-user salt for password hashing**: Security gap — all passwords use the same salt.
- **Invoice numbering gap prevention**: Concurrent creates could result in duplicate numbers (no DB transaction).
- **Address fields in settings**: City, state, postalCode, country not bound in the settings form (line1 only).
